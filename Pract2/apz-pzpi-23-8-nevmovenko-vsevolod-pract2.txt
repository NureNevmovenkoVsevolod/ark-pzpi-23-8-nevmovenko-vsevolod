2 ЗАВДАННЯ
	Метою даної роботи є глибоке вивчення та практичне засвоєння фундаментальних методів рефакторингу програмного коду, описаних у класичній праці Мартіна Фаулера «Refactoring: Improving the Design of Existing Code». Завдання полягає у виборі трьох специфічних методів та демонстрації їх ефективності на прикладі реального коду, що відповідає сучасним стандартам якості та читабельності. Обрані для дослідження методи рефакторингу:
- Replace Magic Number with Symbolic Constant (Заміна магічного числа символьною константою) — для підвищення семантичної зрозумілості коду та полегшення його підтримки;
- Replace Nested Conditional with Guard Clauses (Заміна вкладених умовних операторів граничними операторами) — для спрощення складної логіки та усунення надмірної вкладеності (так званого «ефекту піраміди»);
- Replace Temp with Query (Заміна тимчасової змінної запитом) — для декомпозиції функцій та зменшення кількості локальних змінних, що ускладнюють розуміння контексту.
	Для кожного методу необхідно обов’язково:
    1. Навести теоретичне обґрунтування його доцільності;
    2. Представити лістинг коду «до» рефакторингу з аналізом недоліків (дублювання, низька читабельність, складність підтримки);
    3. Представити лістинг коду «після» рефакторингу з використанням нумерації рядків;
    4. Провести порівняльний аналіз, пояснивши переваги застосованого рішення для архітектури програмного забезпечення.


3 ОПИС ВИКОНАНОЇ РОБОТИ
	Рефакторинг у середовищі OpenResty має важливе значення. Застосування методів рефакторингу дозволяє зробити серверну логіку більш структурованою, спростити підтримку програмного забезпечення, а також знизити ймовірність виникнення помилок у процесі подальшого розвитку проєкту. Крім того, дотримання стандартів OpenResty сприяє написанню коду, який краще піддається оптимізації з боку JIT-компілятора, що позитивно впливає на продуктивність. У межах даної теми розглядаються три поширені методи рефакторингу, які відповідають принципам оформлення та організації коду в OpenResty. Їх використання є доцільним під час розробки серверних модулів, обробки HTTP-запитів та реалізації бізнес-логіки веб-додатків на Lua.

3.1 Метод Replace Magic Number with Symbolic Constant
	У Lua-скриптах для Nginx часто використовуються «магічні числа» для позначення статусів HTTP, лімітів пам'яті або часу очікування. Використання іменованих констант дозволяє уникнути помилок при зміні параметрів системи та робить код самодокументованим.




Код ДО рефакторингу:
1 local function handle_request()
2     local status = get_service_status()
3     -- 200 — успіх, 3600 — час кешування в секундах
4     if status == 200 then
5         ngx.header["Cache-Control"] = "max-age=3600"
6         return ngx.exit(200)
7     end
8 end

	Числа 200 та 3600 не мають семантичного значення. Якщо політика кешування зміниться, розробнику доведеться шукати це число по всьому проекту.

Код ПІСЛЯ рефакторингу:
1 local HTTP_OK = ngx.HTTP_OK
2 local DEFAULT_CACHE_TTL = 3600 -- 1 година в секундах
3 
4 local function handle_request()
5     local status = get_service_status()
6     
7     if status == HTTP_OK then
8         ngx.header["Cache-Control"] = "max-age=" .. DEFAULT_CACHE_TTL
9         return ngx.exit(HTTP_OK)
10    end
11 end

Використання констант (рядки 1-2) робить код зрозумілим без коментарів. Зміна часу кешування тепер виконується в одному місці, що відповідає вимогам стандарту OpenResty щодо підтримки коду.

3.2 Метод Replace Nested Conditional with Guard Clauses
	Глибока вкладеність умов if-then-else ускладнює розуміння бізнес-логіки обробки запиту. Метод «Граничного оператора» (Guard Clause) дозволяє перевірити всі помилкові стани на початку функції та негайно вийти, залишаючи основний успішний сценарій на верхньому рівні вкладеності.

Код ДО рефакторингу:
1 local function process_user_data(data)
2     if data then
3         if data.id then
4             if data.is_active then
5                 save_to_db(data)
6                 return true
7             else
8                 return nil, "user inactive"
9             end
10        else
11            return nil, "no id"
12        end
13    else
14        return nil, "no data"
15    end
16 end

	«Ефект піраміди» (рядки 2-5) робить код важким для сприйняття. Основна дія (save_to_db) заблокована глибоко всередині.

Код ПІСЛЯ рефакторингу:
1 local function process_user_data(data)
2     if not data then 
3         return nil, "no data" 
4     end
5     
6     if not data.id then 
7         return nil, "no id" 
8     end
9     
10    if not data.is_active then 
11        return nil, "user inactive" 
12    end
13
14    save_to_db(data)
15    return true
16 end

	Код став лінійним. Кожна перевірка (рядки 2, 6, 10) працює як фільтр. Це значно знижує когнітивне навантаження та спрощує додавання нових перевірок у майбутньому.

3.3 Метод Replace Temp with Query
	Тимчасові змінні, що зберігають результати складних обчислень (наприклад, розрахунок хешу або формування заголовка), можуть захаращувати функцію. Винесення такої логіки в окремі невеликі функції-запити дозволяє декомпозувати код та полегшити тестування.

Код ДО рефакторингу:
1 local function send_response(content)
2     local content_length = #content -- Тимчасова змінна
3     local signature = ngx.md5(content .. secret_key) -- Тимчасова змінна
4     
5     ngx.header["Content-Length"] = content_length
6     ngx.header["X-Signature"] = signature
7     ngx.say(content)
8 end

Код ПІСЛЯ рефакторингу:
1 local function get_signature(str)
2     return ngx.md5(str .. secret_key)
3 end
4 
5 local function send_response(content)
6     ngx.header["Content-Length"] = #content
7     ngx.header["X-Signature"] = get_signature(content)
8     
9     ngx.say(content)
10 end

	Видалення тимчасових змінних дозволило виділити логіку підпису в окрему функцію get_signature (рядок 1). Це покращує модульність коду та дозволяє перевикористовувати логіку підпису в інших частинах проекту без дублювання.

	3.4 Приклад комплексного рефакторингу
	У попередніх пунктах методи розглядалися ізольовано. Проте в реальних умовах розробки під OpenResty рефакторинг часто вимагає комбінування кількох підходів для перетворення заплутаного («спагетті») коду на чисту та швидку структуру, що не блокує Nginx.
Розглянемо функцію process_api_request, яка перевіряє права доступу, ліміти запитів та формує підпис відповіді.

Код ДО рефакторингу:
1  -- Функція-обгортка, яку треба прибрати (Inline)
2  local function get_raw_key(req) return req.api_key end
3
4  local function process_api_request(req)
5      -- Зчитуємо ключ через зайву функцію
6      local key = get_raw_key(req)
7
8      -- Складна умова та магічні числа (Guard Clauses + Symbolic Constant)
9      if key and #key == 32 then
10         if ngx.shared.limit_store:get(key) < 1000 then
11             -- Логіка, яку варто винести (Extract Method / Temp with Query)
12             local sig = ngx.md5(key .. secret_salt)
13             ngx.header["X-Auth-Sig"] = sig
14             return ngx.exit(200)
15         else
16             return ngx.exit(429)
17         end
18     else
19         return ngx.exit(403)
20     end
21 end

	Застосовані зміни:
    1. Inline Method: Видалено функцію get_raw_key, звернення йде напряму до поля req.api_key.
    2. Replace Magic Number with Symbolic Constant: Числа 1000, 200, 429 замінено на зрозумілі константи.
    3. Replace Nested Conditional with Guard Clauses: Вкладені if замінено на плоску структуру перевірок.
    4. Replace Temp with Query: Розрахунок підпису винесено в окрему функцію-запит.

Код ПІСЛЯ рефакторингу:
1  local HTTP_OK = ngx.HTTP_OK
2  local HTTP_FORBIDDEN = ngx.HTTP_FORBIDDEN
3  local HTTP_TOO_MANY_REQUESTS = 429
4  local RATE_LIMIT = 1000
5
6  -- 3. Temp with Query: винесена логіка генерації підпису
7  local function get_auth_sig(key)
8      return ngx.md5(key .. secret_salt)
9  end
10
11 local function process_api_request(req)
12     -- 1. Inline: прямий доступ до ключа
13     local key = req.api_key
14
15     -- 2. Guard Clauses: лінійна перевірка умов
16     if not key or #key ~= 32 then
17         return ngx.exit(HTTP_FORBIDDEN)
18     end
19
20     if ngx.shared.limit_store:get(key) >= RATE_LIMIT then
21         return ngx.exit(HTTP_TOO_MANY_REQUESTS)
22     end
23
24     -- Успішний сценарій на верхньому рівні
25     ngx.header["X-Auth-Sig"] = get_auth_sig(key)
26     return ngx.exit(HTTP_OK)
27 end

	3.5 Покрокова демонстрація процесу рефакторингу
	Рефакторинг в OpenResty дисциплінована техніка, де кожна зміна має бути верифікована. Правильний процес виглядає так:
    1. Перевірка наявності тестів: Перед початком змін необхідно впевнитися, що поточний код проходить інтеграційні тести. В OpenResty стандартним інструментом є Test::Nginx (запуск тестів, що емулюють HTTP-запити).
    2. Вибір методу: Визначаємо "запахи коду" (code smells) — наприклад, використання магічного числа 429 або заплутані вкладені if.
    3. Застосування змін малими кроками:
- Замінили магічне число константою;
- Перевірили, чи стартує Nginx (nginx -t);
- Застосували Guard Clause для однієї з умов;
- Запустили тести та перевірили error.log на наявність помилок Lua.
    4. Перевірка стилю: Запуск утиліти luacheck, щоб переконатися, що не було створено випадкових глобальних змінних (що ми обговорювали в ПЗ №1) та дотримано відступів.
    5. Коміт змін: Фіксація результату в Git з чітким описом застосованого методу рефакторингу.


3.6 Інструменти для рефакторингу в C/Linux
	На відміну від статично типізованих мов, Lua вимагає специфічних інструментів для безпечного рефакторингу:
    1. Luacheck: Статичний аналізатор коду. Це головний інструмент для рефакторингу в Lua. Він миттєво знаходить "сміття" — невикористані змінні (після Inline Method) або випадкове забруднення глобальної області видимості.
    2. Test::Nginx: Фреймворк для тестування OpenResty-модулів. Він дозволяє автоматизувати перевірку того, що після рефакторингу логіка обробки HTTP-запитів залишилася незмінною.
    3. Luanalysis / EmmyLua: Сучасні плагіни для IDE (IntelliJ IDEA / VS Code), які додають статичну типізацію через анотації. Це дозволяє безпечно виконувати операцію "Rename Symbol" (перейменування функцій та полів таблиць) у великих проектах.
    4. OpenResty CLI (resty): Інструмент для швидкого виконання фрагментів Lua-коду. Дозволяє миттєво перевірити ізольовану логіку нової функції (наприклад, після застосування "Extract Method") без повного перезапуску Nginx.


4 ВИСНОВКИ
Під час виконання даної практичної роботи було проведено глибокий аналіз та практичне застосування методів рефакторингу програмного коду за класифікацією Мартіна Фаулера у контексті високопродуктивної розробки на платформі OpenResty.
	Досліджено три фундаментальні техніки: Replace Magic Number with Symbolic Constant, що дозволило позбутися незрозумілих числових літералів на користь семантичних констант; Replace Nested Conditional with Guard Clauses, що забезпечило лінійну структуру коду та усунуло надмірну вкладеність умов; а також Replace Temp with Query, що сприяло декомпозиції функцій та покращенню їхньої модульності. Окрему увагу було приділено тому, як рефакторинг впливає на роботу LuaJIT, зокрема через зменшення кількості тимчасових змінних та спрощення логічних переходів.
	Встановлено, що застосування цих методів не є простою формальністю, а виступає критичним інструментом для забезпечення довгострокової підтримки проекту, підвищення читабельності алгоритмів та мінімізації помилок при масштабуванні систем реального часу.
